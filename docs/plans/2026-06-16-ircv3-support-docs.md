# Design: IRCv3 Support Documentation

**Date:** 2026-06-16
**Status:** Approved (design phase)
**Topic:** Document every IRCv3 capability Cascade supports and how each surfaces in the client.

## Goal

Produce a single, layered reference — `docs/ircv3-support.md` — that documents Cascade's
IRCv3 support for two audiences at once:

- **End users / evaluators:** plain-language "what you get" framing at the top.
- **Contributors:** a full per-capability technical reference with `file:line` pointers
  and screenshots of the real UI.

The doc doubles as a **roadmap**: a complete spec-compliance matrix lists every notable
IRCv3 capability with its status, including the ones not yet implemented.

## Decisions (from brainstorming)

| Question | Decision |
|----------|----------|
| Audience | **Both, layered** — user-facing intro + contributor reference in one file |
| Scope | **Full spec matrix** — every notable IRCv3 cap with status, not just supported ones |
| "How it shows up" | **Descriptions + screenshots** |
| Screenshot source | **Playwright e2e tests** — committed images regenerated by real test runs |
| Location | `docs/ircv3-support.md` (doc); design doc here under `docs/plans/` per repo convention |

## Deliverable structure — `docs/ircv3-support.md`

### Section 1 — Overview / "What you get" (user-facing)
Plain language, no code refs: secure login (SASL), accurate timestamps (server-time),
durable scrollback across reconnects (chathistory + batch + msgid), no duplicate messages
(echo-message).

### Section 2 — Capability status matrix (the full spec matrix)
Table columns: **Capability · Status (✅ Supported / ◐ Partial / ⛔ Not yet) · Negotiated? ·
Note**. Covers supported, supporting, and not-yet sets in one place.

### Section 3 — Per-capability detail (contributor reference, supported caps)
One subsection per supported capability:
1. What the spec does (1–2 sentences)
2. How Cascade negotiates it (`client.go:25` + conditions)
3. Backend behavior (`file:line`)
4. **How it shows up in the client** — prose + component `file:line` + screenshot

### Section 4 — Not-yet-supported notes
Short rationale/blocker per absent cap (e.g. presence family needs roster state tracking),
so the matrix reads as a roadmap.

## Screenshot workflow (Playwright e2e)

Screenshots are produced by dedicated Playwright specs and committed as image files.

- **New spec(s):** `e2e/tests/screenshots-ircv3.spec.ts` (split per feature if it grows).
- **Boot:** existing harness — `page.goto(runtime.bridgeUrl)`, helpers from `e2e/lib/actions.ts`
  (`addNetworkAndConnect`, `selectNetwork`, `joinChannel`).
- **Traffic:** `e2e/lib/irc-peer.ts` `IrcPeer` against the Docker Ergo server
  (`runtime.ergoPort`). Ergo supports the caps Cascade negotiates, so behavior is real.
- **Capture:** explicit `await page.screenshot({ path: ... })` /
  `locator.screenshot()` (current config is failure-only).
- **Committed location:** `docs/images/ircv3/<capability>.png`; referenced from the doc.
- **Determinism:** mask or freeze non-deterministic regions (server-time clock values,
  idle/sign-on counters) so committed images are stable across runs. Decide masking vs.
  fixed-input approach during implementation.

Run: `task e2e` (full) or `cd e2e && npx playwright test tests/screenshots-ircv3.spec.ts`.

## Implementation phases (for the plan)

1. **Write the doc text** — Sections 1, 2, 4 and the prose/`file:line` parts of Section 3.
   Verify every capability claim and line reference against current code. Commit.
2. **Screenshot specs** — add `e2e/tests/screenshots-ircv3.spec.ts` producing one image per
   visible capability; solve determinism; commit images under `docs/images/ircv3/`.
3. **Wire images into the doc** — embed screenshots in Section 3; final pass for accuracy.

## Non-goals

- No new IRCv3 capabilities are implemented as part of this work — documentation only.
- No refactor of the IRC client or e2e harness beyond adding screenshot specs.
- Not user-help/onboarding docs; this is a capability reference + roadmap.
