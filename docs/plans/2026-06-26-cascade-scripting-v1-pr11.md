## Task 3: Writing scripts page

**Files:**
- Create: `docs/public/scripting/writing-scripts.md`
- Modify: `zensical.toml` (append `scripting/writing-scripts.md`)

- [ ] **Step 1: Create the Writing-scripts page**

Write `docs/public/scripting/writing-scripts.md`. Content outline:
- **"A script is a directory"**: a folder of `package main` `.go` files under the scripts dir; the folder name is the script's ID. Multiple files are merged at load; all must be `package main`. Subfolders are ignored.
- **"The manifest header"**: `// cascade:` comment lines parsed from any file. Recognized keys (exact): `name` (display name; defaults to the folder name), `description` (free text), `permissions` (comma-separated). Example:

```go
package main

// cascade:name My Greeter
// cascade:description Greets people who say !hi
// cascade:permissions network

import "github.com/matt0x6f/irc-client/cascade"
```

  Note plainly: **permissions are recorded and shown in the panel but not enforced in v1** — declare them for forward-compatibility; enforcement comes later.
- **"Handlers"**: the five optional exported functions and when each fires (exact signatures — keep consistent with [API reference](api-reference.md)):
  - `func Setup(c *cascade.Client)` — runs once when the script loads (and on every reload/enable); use it for timers and proactive setup.
  - `func OnText(e cascade.TextEvent)` — an inbound channel/DM message (PRIVMSG).
  - `func OnNotice(e cascade.NoticeEvent)` — an inbound NOTICE.
  - `func OnJoin(e cascade.JoinEvent)` — someone joined a channel.
  - `func OnPart(e cascade.PartEvent)` — someone left a channel.
  Define any subset. Cascade never delivers your own messages back to you.
- **"Reading & replying to messages"**: `e.Nick`, `e.Channel`, `e.Message`; helpers `e.HasPrefix("!cmd")`, `e.Arg(n)` (1-based whitespace fields, `""` out of range), `e.IsDM()`; `e.Reply(text)` routes to the channel (or the sender for a DM). A small inline example using `Arg`:

```go
package main

// cascade:name echo

import "github.com/matt0x6f/irc-client/cascade"

func OnText(e cascade.TextEvent) {
	if e.Arg(1) == "!echo" {
		e.Reply(e.Message)
	}
}
```

- **"Acting on your own — `Setup`, timers, and `Say`"**: `Setup(c)` receives a `*cascade.Client`. `c.Network("<name>")` returns a handle; `.Say(target, message)` sends a PRIVMSG. `c.Every("30m", fn)` repeats; `c.After("10s", fn)` fires once. Durations are Go duration strings; an invalid one is ignored. Timers are stopped automatically on reload/disable. Example:

```go
package main

// cascade:name standup

import "github.com/matt0x6f/irc-client/cascade"

func Setup(c *cascade.Client) {
	c.Every("1h", func() {
		c.Network("Libera").Say("#team", "hourly check-in")
	})
}
```

- **"The sandbox — what you can and can't import"**: prominent callout. You can import **only** `github.com/matt0x6f/irc-client/cascade`. There is **no standard library** — not `fmt`, not `strings`, nothing. Use the event helpers (`HasPrefix`, `Arg`) and Go built-ins (string `+`, `len`, `append`, `switch`, `range`, maps/slices). Importing anything else makes the script fail to load with an **Error** status. Link to [Lifecycle & limits](lifecycle-and-limits.md).
- **"Editor setup"**: Cascade scaffolds a `go.mod` (`module cascade-scripts`, `go 1.21`, requiring `…/cascade v1.0.0`) so your editor + gopls give real autocomplete and type-checking. **Caveat (v1):** gopls can only resolve the `cascade` import once the `cascade/v1.0.0` module tag is published; until then, add a local `replace` to your scripts-dir `go.mod` pointing at a Cascade checkout, e.g. `replace github.com/matt0x6f/irc-client/cascade => /path/to/irc-client/cascade`. This affects only editor tooling — the running app interprets the script regardless.
- End: "See [Examples](examples.md) for complete recipes and the [API reference](api-reference.md) for every type and method."

type PartEvent struct {

**Files:**

- [ ] **Step 1: Create the Lifecycle-and-limits page**

Write `docs/public/scripting/lifecycle-and-limits.md`. Content outline:
- **"Lifecycle"**: load → run → hot-reload on save → enable/disable → unload (delete the folder). On reload/enable, `Setup` re-runs and timers are recreated. Note plainly: **package-level variables reset on reload** — don't rely on in-memory state surviving a save.
- **"Status values"**: a table of the four exact statuses and what each means:
  - `loaded` — running normally.
  - `disabled` — turned off (by you in the panel, or persisted off across restarts).
  - `error` — failed to load (compile/type error, a forbidden import, or `Setup` panicked). The panel shows the message.
  - `runaway` — auto-disabled by the watchdog (see below).
- **"The watchdog (keep handlers fast)"**: every handler/timer call is bounded to **2 seconds**. Exceed it and the script is auto-disabled as `runaway` — Cascade contains the damage (stops further dispatch) but cannot kill an already-spinning goroutine, so **never block, sleep-loop, or run unbounded work in a handler**. Separately, **3** consecutive failing dispatches (e.g. panics) also auto-disable the script as `runaway`; a single success resets the counter. Re-enable from the panel (it clears the strike count and re-runs `Setup`).
- **"The sandbox"**: scripts can import **only** `cascade` — no stdlib, no filesystem, no direct network. This is the security model: a script can do exactly what the `cascade` API allows and nothing more. Importing anything else is a load-time `error`. (Cross-link back to [Writing scripts](writing-scripts.md).)
- **"Permissions (v1)"**: `// cascade:permissions a, b` is parsed and displayed in the panel but **not enforced yet** — it's a forward-looking declaration. Don't treat it as a security boundary in v1.
- **"Editor tooling caveat"**: repeat the gopls/tag note from Writing scripts briefly, with the `replace` workaround, and clarify it's editor-only (runtime is unaffected).
- **"When to use a plugin instead"**: short pointer — if you need real isolation, third-party dependencies, or to run untrusted/hostile input, graduate to a [plugin](../developers/plugin-system.md).

Create `e2e/tests/screenshots-scripting.spec.ts`. It must: be gated on `CASCADE_SCREENSHOTS`; open the Scripts settings pane; create a script via the panel so the screenshot shows a populated, real panel; then screenshot the panel into `docs/public/scripting/images/`.

```ts
import * as fs from 'fs';
import * as path from 'path';
import { test, expect } from '../lib/fixtures';
import { openSettings } from '../lib/actions';

/**
 * Generates the screenshots embedded in docs/public/scripting/managing-scripts.md.
 *
 * Documentation artifacts, not assertions — only run when explicitly requested
 * via CASCADE_SCREENSHOTS=1 so the normal e2e/CI run doesn't rewrite tracked
 * PNGs. Regenerate intentionally:
 *
 *   cd e2e && CASCADE_SCREENSHOTS=1 npx playwright test tests/screenshots-scripting.spec.ts
 *
 * Images land in docs/public/scripting/images/ (two levels up from this spec).
 */

const OUT_DIR = path.resolve(__dirname, '../../docs/public/scripting/images');

test.describe('Scripting documentation screenshots', () => {
  test.skip(!process.env.CASCADE_SCREENSHOTS, 'set CASCADE_SCREENSHOTS=1 to regenerate doc screenshots');

  test.beforeAll(() => {
    fs.mkdirSync(OUT_DIR, { recursive: true });
  });

  test('Scripts panel with a created script', async ({ page, runtime }) => {
    // openSettings opens a new page at ?view=settings&section=scripts.
    const settings = await openSettings(page, runtime, 'scripts');

    // Create a script so the panel is populated (not the empty state).
    await settings.getByPlaceholder(/script name/i).fill('greeter');
    await settings.getByRole('button', { name: /^new script$/i }).click();

    // Wait for the row to appear (hot-reload loads the new script).
    await expect(settings.getByText('greeter')).toBeVisible({ timeout: 15_000 });

    await settings.getByTestId('scripts-panel').screenshot({
      path: path.join(OUT_DIR, 'scripts-panel.png'),
    });
  });
});
```
